SYSTEM ARCHITECTURE · 架構與運作全解

ailiveX
運作原理全解析

一個以「用戶」為中心的 AI 角色記憶 × 即時語音平台。每個用戶 × 每個角色各自記憶、不共享——同一個角色對不同人，記得的事不一樣。本文件完整拆解三條核心管線、語音雙腦機制、記憶系統，以及所有依賴的第三方服務。

00

系統定位

ailiveX 是一個 AI 角色互動平台。使用者與被指派的 AI 角色進行文字對話或即時語音通話，角色會記住這個用戶（記憶系統），並能在對話中代為生成策略書、企劃書等正式文件。

整個架構的核心翻轉是：從「以角色為中心」改成「以用戶為中心」。記憶、對話、文件一律嚴格綁定 (userId, characterId) 這一對；角色不共享記憶——同一個角色面對 A 與 B，記得的事完全分開。

兩種身份

用戶 / 管理員

用戶走前台（大廳→對話），只看到被指派的角色；管理員走後台，管角色、帳號、權限、記憶。

三條管線

文字 · 語音 · 文件

文字對話、即時語音通話、非同步文件生成——三條獨立但共用同一份記憶與靈魂。

一個靈魂

soul → soulCore

管理員寫角色「靈魂」原文，系統鑄煉成高密度 soulCore，注入每一次對話。

01

系統總覽

系統分成三個運算層與一個資料層，背後串接多個外部 AI 服務。前端與所有 API 跑在 Vercel；即時語音的「大腦」是一支獨立的 Python Agent，跑在 Google Cloud Run；資料的單一真相源是 Firestore。

前端 Next.js 16 · React 19 語音 Python · LiveKit Agents 1.5.1 資料庫 Firestore 儲存 Cloud Storage 區域 asia-east1 / us-central1
02

使用者身份與頁面流程

登入後依身份分流：管理員進後台，用戶進大廳。用戶在大廳挑角色，進入文字聊天頁，再從那裡可升級成即時語音通話；也能查看角色幫他生成的文件。

03

核心理念：以用戶為中心的記憶

這是整個系統的靈魂。所有記憶都被綁在 (userId, characterId) 這一對鑰匙上。同一個角色，對不同用戶各記各的；角色因此能像「記得你的老朋友」，而不是「拿著同一張備忘錄的陌生人」。

設計哲學

活著的人有時間感、有關係弧線、會記得情緒、會追問未完成的事。記憶系統 v2 的目標，就是讓角色具備這四件事——每條記憶都帶相對時間（今天 / X 天前 / X 個月前），並追蹤「我們認識多久、聊過幾次」。

每次對話結束，系統會非同步用 LLM 從逐字稿中提煉記憶，分成六種類型存進 Firestore；下次對話開場前，再把最相關的記憶組成有層次的 system prompt 注入角色。細節見 第 08 節。

04

管線 A · 文字對話

文字對話走 Vercel 的 /api/dialogue 路由。一次請求的完整流程：驗證身份與權限 → 載入靈魂、記憶、歷史 → 呼叫 Claude → 用程式解析工具標記 → 回覆並存檔。記憶提煉與文件派工放在回覆送出之後非同步執行，避免拖慢回應。

關鍵設計

  • 工具靠文字標記，不靠 tool_use：因為走 bridge 省錢、而 bridge 不支援原生 tool calling，所以角色用 [[REMEMBER]]…[[/REMEMBER]]、[[DOCUMENT title="…"]]…[[/DOCUMENT]] 文字標記，再由程式確定性解析（天條：解析/格式是程式的工作，不丟 LLM）。
  • 後端把關權限：未被指派該角色 → 直接 403，不只靠大廳隱藏。
  • 讀網址：訊息含 URL 時，伺服器抓網頁正文餵進這一輪 context（有 SSRF 防護：擋私有 IP、雲端 metadata、逐跳重驗 redirect）。
  • 非同步收尾：記憶提煉 + 關係更新 + 文件派工都放進 after()，回覆已先送出，使用者不等。
05

管線 B · 即時語音通話

即時語音是系統最複雜的部分。瀏覽器先向後端要一張 LiveKit token，token 裡夾帶角色資訊與一個 RoomAgentDispatch，指名要哪一版的 Python Agent 進房。Agent 進房後從 Firestore 讀出角色靈魂與記憶，組成 system prompt，然後跑一個「聽 → 想 → 說」的即時迴圈。

關鍵設計

  • 靠 agent_name 隔離：token 用 RoomAgentDispatch 指名 ailivex-realtime-v10，與其他版本、甚至其他專案共用同一個 LiveKit project 也不衝突。房名一律 ailivex- 前綴，Agent 端會拒絕沒前綴的房（安全把關）。
  • 語音主迴圈用直連 key：即時對話需要串流，而 bridge 不支援串流也不支援 tool_use，所以主說話的 Sonnet 一律直連付費 ANTHROPIC_API_KEY。
  • 掛斷不丟記憶：shutdown 視窗 90 秒，容得下掛斷後的 LLM 提煉；先秒存逐字稿，再並行萃取「上次對話快照（lastSession）」與長期記憶。
  • 接得上上次：開場第一句會優先看「上次聊到最後的原話」，從沒聊完的事接回來，而不是念摘要。
06

語音版本演進 · v1 → v10

即時語音是實驗性的，會直接改變體感。所以團隊用一條鐵律：每一版完全隔離——獨立的 agent_name、獨立的 Cloud Run 服務、獨立的前端路由、獨立的 build 設定。實驗版絕不影響穩定版，回滾就是「不要把流量切到新版」。

重要 · 版本紀律

所有版本共用同一個 Docker image，靠啟動命令 python -m agent.main_vN start 區分。要做新實驗，複製成新的 vN，絕不直接改現役版本。沒有 v7（版本從 v6 直接跳到 v8）。README 已過時（只寫到 v4），現役以程式與 git 為準。

07

多人房雙腦機制（v10）

當一個焦點 AI 要在「一群真人」裡像真人一樣參與，挑戰是：什麼時候該開口、什麼時候該閉嘴、誰在跟誰說話。v10 用雙腦分工解決——一顆腦負責「判斷」，一顆腦負責「開口」。天條是：判斷可以丟 LLM，但「要不要說」的開關由程式決定，機制不丟 LLM。

判斷腦 · 背景

Claude Haiku 4.5

floor-gate 發言權判斷、學習講者名冊、評估內心狀態（stance / activation / want_to_speak）。便宜、快、跑在對話路徑之外。

開口腦 · 主說

Claude Sonnet 4.6

真正生成要說出口的話、搶話時的回應。較慢、較貴、品質高。只在判斷腦放行後才動用。

v10 三補強

  • ① 回音過濾：角色自己的 TTS 被麥克風收回、被標成「旁邊另一位」時，會與自己近期輸出比對，高度吻合就直接丟——不判斷、不記憶、不算成另一個講者。這根治了「角色對自己讓位、把自己的話記成別人說的」的萬惡之源。
  • ② 講者身份名冊：逐字稿帶 diarization 標記 #N；背景判斷腦兼差學名字（有人自我介紹或被點名 → #2 = 名字），讓角色分得清誰是誰。
  • ③ 3a 收斂：一對一時維持原樣（陪伴、填冷場）；多人時不再因為「冷場」就開口，改成「只有判斷腦說我真的有話要加（want_to_speak）才講」，避免 AI 變成主持人。
08

記憶系統細節

記憶分六種類型、三層分級，組成七區塊的 system prompt。寫入時用向量去重，讀取時用語義排序，並有「過期（stale）」與「主動回想（active recall）」機制。

讀寫機制

  • 提煉：對話結束後用 Haiku 從逐字稿提取 1–2 條記憶，判給六種 type 之一，附 importance 1–10。
  • 去重：寫入前算 Vertex embedding，與既有記憶餘弦相似度 ≥ 0.85 視為重複，跳過。
  • 晉升：fresh 被帶進對話命中 3 次 → 自動升 core。
  • 主動回想：question 類超過 7 天沒解決，會自然帶進「懸而未決」區塊，讓角色主動追問。
  • 過期：question > 60 天、emotion > 90 天 → 讀取時 lazy 標成 stale，不再注入。
  • 語義排序：有當前訊息時，fact 區塊用 embedding 相似度排序，挑最相關的注入。
09

管線 C · 文件生成

當用戶在對話中請角色「寫一份策略書/企劃書」，角色會發出 [[DOCUMENT]] 標記（語音則呼叫 write_document 工具）。系統建立一筆 job + document 紀錄，非同步派工給 worker 生成，用戶可在「我的文件」看到狀態並下載。

兩條 worker 路徑

主路徑＝ Vercel 的 /api/doc-process 路由（走 bridge 吃到飽，生成失敗回非重試狀態避免無限重排）。Cloud Run 的 doc-worker（Express，/process）為次要/舊路徑；原本的 Cloud Tasks 派工鏈已廢棄（enqueue.ts 在 env 未設時 no-op）。語音 Agent 端則用背景 thread 直接 POST 到 DOC_WORKER_URL。

10

資料模型 · Firestore

Firestore 是單一真相源，定義在 src/lib/collections.ts。核心不變量：memories / conversations / documents 一律嚴格綁 (userId, characterId)，多數對應文件用確定性 docId ${userId}_${characterId}。

Collection用途關鍵欄位
users帳號username, passwordHash(scrypt), displayName, role
charactersAI 角色soul → soulCore, avatarUrl, voiceIdMinimax, voiceSettings, convSettings, aliases
access權限指派（allowlist）docId = userId_characterId（存在即有權）
conversations文字對話歷史messages[], summary, messageCount；Agent 另寫 lastSession
memories記憶content, embedding, importance, tier, type, status, hitCount
relationships關係追蹤conversationCount, first/lastConversationAt
documents生成的文件title, mdContent, htmlUrl, status
jobs文件生成任務brief, documentId, status
config/globalPrompts全域 Prompt 結點antiSycophancy, timeRule, abilities, voiceRules
zhu_vitals_costLLM 用量成本紀錄model, tokens, cost_usd_est, route（90 天 TTL）

認證與權限

密碼

scrypt

Node 內建 scrypt，格式 salt:hash（hex），比對用 timing-safe。

Session

HMAC Cookie

httpOnly ailivex_session，Web Crypto 簽章（Edge + Node 都能跑），30 天，無狀態。

把關

middleware + hasAccess

middleware 擋未登入、限制 /admin；後端每條路徑再查 access 指派，不靠前端隱藏。

公開路徑只有 /login、/api/auth/login、/api/doc-process（後者靠 x-worker-secret 保護）。/admin* 與 /api/admin* 限管理員。

LLM 路由 · bridge vs 直連

系統的 Claude 呼叫走兩條路，目的是省錢與穩定的取捨。zhu-bridge 把 API 計費轉成 Claude Max 月費「吃到飽」，但不支援串流、也不支援原生 tool_use。所以：

注意

切換靠 BRIDGE_ENABLED + BRIDGE_URL + BRIDGE_SECRET。若三者齊備就走 bridge；bridge 執行期失敗一律 throw、不退回 SDK（避免雙重燒錢）。設定不全時才退回直連 SDK。
