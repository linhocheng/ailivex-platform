# AI 多角色 Podcast 生成系統 — 技術白皮書

**版本**：v1.0　**日期**：2026-07-01　**平台**：ailiveX

---

## 一、系統概述

本系統讓使用者選擇 2 位以上 AI 角色，輸入對話主題與討論焦點，自動生成一集多人 Podcast 腳本，再將腳本轉換為語音音檔輸出。

整體流程分為兩個階段：

```
[使用者輸入]
  對話主題 / 討論焦點 / 目標字數 / 角色選擇
        ↓
[Phase 1：腳本生成]
  LLM 代入每個角色 → 生成多行對話腳本
        ↓
[使用者可手動編輯腳本]
        ↓
[Phase 2：音檔生成]
  每個角色為自己的台詞加情緒標記
  → 逐行 TTS（MiniMax speech-2.8-hd）
  → 合併 MP3 → 上傳 GCS → 回傳音檔 URL
```

## 二、角色進場

### 2.1 角色資料來源

每個角色在系統中有兩個欄位承載個性：

| 欄位 | 說明 |
|------|------|
| `soul` | 角色原始靈魂文字，由管理員手寫 |
| `soulCore` | 經 LLM 精煉後的高密度版本，優先使用 |

進場時，系統會取 `soulCore`（若存在）否則退回 `soul`，直接注入 system prompt 的最前方。**角色個性是 prompt 的第一層，結構規則是第二層**，確保靈魂優先於指令。

### 2.2 主持人與來賓的區分

使用者在 UI 選擇角色時，**第一位選擇的角色自動成為主持人，其餘為來賓**。

這個區分只影響一件事：**誰負責切入話題**。主持人的職責是用第一句話把對話方向拋出來，然後退回自己的個性。來賓不需要等待介紹，直接以自己的立場回應。

### 2.3 角色注入格式

```
【主持人】{name}
{soulCore 或 soul}

【來賓】{name}
{soulCore 或 soul}
```

兩個角色的靈魂文字並列注入，讓 LLM 同時持有兩個視角，在生成時能真正對應彼此，而不是各說各話。

## 三、Phase 1：腳本生成 Prompt 設計

### 3.1 System Prompt 結構

```
[角色靈魂注入區]        ← soulCore / soul，角色個性基礎
[對話規則]              ← 什麼可以做、什麼明確禁止
[對話走向]              ← 三段結構（前/中/後），只給方向不給台詞
[輸出格式]              ← [角色名]: 台詞，純文字，無說明
```

### 3.2 對話規則（現行版本）

```
· 主持人的職責是把話題引出來、推動對話往焦點走——不是主持節目，是在聊天。
· 其他角色自由回應：認同就認同，不認同就直說，不需要轉彎、不需要客氣。
· 對話可以有碰撞——觀點有時一致、有時對立、有時延伸——這才是真實的交流。
· 禁止：「歡迎收聽」「非常榮幸邀請到」「謝謝你來」這類節目儀式台詞。
· 禁止：「讓我們先確認一下今天的主題是……」這種流程確認句。
· 如果角色需要說出自己在做什麼，讓它自然發生在對話中，不是走自我介紹儀式。
```

**設計邏輯**：規則以「禁止」為主，而非「應該做什麼」。原因是 LLM 在收到「應該做自我介紹」這類指示時，會把指示執行得很完整但失去個性；收到「禁止某行為」時，靈魂欄位的個性才有空間浮現。

### 3.3 對話走向（三段結構）

```
前段（約 20%）：主持人直接用一句話切入話題，對方接住並表明自己的立場或視角
中段（約 65%）：圍繞焦點來回，每人 1-3 句換人，觀點持續推進，有認同也有反駁
後段（約 15%）：各自說出最重要的一個判斷或結論，對話自然收尾
```

注意：**結構只給比例和方向，不給台詞**。「主持人的第一句話切入話題」是方向，不是「主持人要說：『今天我們來聊 X』」。這個差異決定了角色能否用自己的語氣切入。

### 3.4 User Message 輸入參數

使用者提供三個可選輸入：

| 參數 | 說明 | 範例 |
|------|------|------|
| `對話主題` | 這集聊什麼，給角色定方向 | AI 怎麼改變人的生活型態 |
| `討論焦點` | 必須觸碰到的具體範圍 | 聚焦在如何影響設計師的工作方式 |
| `目標字數` | 腳本總字數，影響集數長度 | 400 / 600 / 900 / 1200 字 |

這三個參數被傳入 user message，不進 system prompt。原因：對話主題和焦點是「這集的任務」，不是角色個性的一部分，放 system 會讓角色把「完成任務」當作主要驅動力，蓋掉個性。

### 3.5 腳本解析（確定性）

LLM 輸出的每一行符合格式：

```
[角色名稱]: 台詞內容
```

系統用正則 `^\[([^\]]+)\][:：]\s*(.+)` 逐行解析，不符合格式的行直接丟棄，不走任何 LLM 修復。解析結果是 `PodcastLine[]`：

```typescript
{ speaker: string; characterId: string; text: string }
```

解析是程式級確定性操作，任何格式錯誤由 LLM 重試解決，不讓 LLM 修 LLM 的輸出。

## 四、Phase 2：情緒標記 Prompt 設計

### 4.1 為什麼要情緒標記

MiniMax TTS 支援在文字中插入語氣和停頓標記，讓聲音更自然：

**語氣標記**：
```
(emm)     — 「嗯……」遲疑猶豫
(breath)  — 輕吸氣，準備說話
(sighs)   — 嘆氣
(chuckle) — 輕笑
(inhale)  — 急促吸氣，驚訝或激動
(exhale)  — 呼氣，釋放情緒
(gasps)   — 驚呼
```

**停頓標記**：
```
<#0.3#>  — 短停 0.3 秒
<#0.5#>  — 中停 0.5 秒
<#1.0#>  — 長停 1 秒
```

### 4.2 每個角色只標自己的台詞

這是關鍵設計：**每個角色獨立收到自己的台詞，用自己的 soulCore/soul 作為 system prompt，為自己的每一行標記**。

```
Character A 的台詞 → 以 A 的 soul 為 system → A 為自己標記
Character B 的台詞 → 以 B 的 soul 為 system → B 為自己標記
```

這樣做的原因：如果所有台詞一起丟給同一個 LLM call，LLM 會從「全知視角」決定標記，標出來的是「這個台詞應該加什麼標記」，而不是「這個角色講這句話時會有什麼語氣」。拆開後，每個角色是從自己的內部視角決定語氣，標記的個性就出來了。

### 4.3 標記 System Prompt

```
{soulCore 或 soul}

以下是你在一場對話裡說的話。請為每一句加上聲音標記，讓語音聽起來像你平時說話的樣子，不是在表演。

規則：
- 台詞的文字內容絕對不能修改，只能插入標記
- 標記要符合你自己的個性，不要硬套、不要過度（一句話最多 1-2 個標記）
- (sighs) 整集最多用 1 次，用太多會讓聽眾覺得你在抱怨
- 輸出格式：每行保留原編號，不加任何說明
```

注意：`不是在表演` 這四個字是刻意的。早期版本說「你正在錄製一集 Podcast」，角色進入「表演模式」，標記選擇偏向刻意演出而非自然流露。

### 4.4 輸入格式（編號對應）

```
1. 我覺得這件事有點複雜
2. 但你說的那個方向我不太認同
3. 等等，你的意思是說……
```

LLM 輸出同格式，用編號確定性對應回原台詞，解析失敗就回退原文，不讓標記破壞腳本。

### 4.5 標記安全防護

LLM 有時會自創不存在的標記（如 `(laugh)` 或 `(pause)`）。系統在接收 LLM 輸出後，用白名單正則過濾：

```typescript
const VALID_INTERJECTION = new Set(['emm', 'breath', 'sighs', 'chuckle', 'inhale', 'exhale', 'gasps'])

function sanitizeTags(text: string): string {
  return text
    .replace(/\(([^)]+)\)/g, (m, inner) =>
      VALID_INTERJECTION.has(inner.trim()) ? m : '')
    .replace(/<([^>]+)>/g, (m, inner) =>
      /^#[\d.]+#$/.test(inner.trim()) ? m : '')
}
```

任何不在白名單的語氣標記直接剝除，停頓標記只保留 `<#數字#>` 格式。這是確定性程式守關，不讓 LLM 的創意輸出破壞 TTS 管道。

### 4.6 換人說話時的自動停頓

角色切換之間，系統自動插入停頓，無需 LLM 介入：

```
同一個人連續說話 → 前插 <#0.3#>（短停）
換另一個人說話  → 前插 <#0.5#>（中停）
第一句台詞      → 不插停頓
```

## 五、TTS 文字正規化

送進 MiniMax 之前，所有台詞經過一套確定性正規化流程：

### 5.1 年份轉換

MiniMax 讀到 `1999` 會唸「一千九百九十九」。系統在 opencc 之前先把帶「年」的四位數年份轉成逐字中文：

```
1999年 → 一九九九年
2024年底 → 二〇二四年底
```

### 5.2 繁體 → 簡體（opencc）

MiniMax 是大陸 TTS，簡體中文的發音更準確。使用 opencc-js `tw → cn` 轉換，並在轉換前保護 MiniMax 標記（用 placeholder 包住，轉換後還原）。

### 5.3 詞彙對齊

opencc 只轉換字形，部分台灣慣用詞轉換後的簡體形式 MiniMax 仍不熟悉，用 Map 補正：

| 台灣原文 | opencc 後 | 送 MiniMax |
|----------|-----------|------------|
| 軟體 | 软体 | **软件** |
| 硬體 | 硬体 | **硬件** |
| 網路 | 网路 | **网络** |
| 晶片 | 晶片 | **芯片** |
| 垃圾 | 垃圾 | **废弃物** |

## 六、已知問題與待優化方向

### 6.1 角色靈魂被儀式語言蓋掉（已修）

早期 prompt 包含「主持人開場白：歡迎聽眾、介紹來賓」「確認彼此身份、建立共識」「格式範例：歡迎收聽，我是……」等，導致角色不論個性如何都走同一套客套儀式。現已全數移除，改為禁令式規則。

### 6.2 靈魂文字本身的品質

`soul` 和 `soulCore` 的品質直接決定對話的個性強度。若 soul 寫得太平、太正式、或帶有 schema 式的欄位描述（如 `identity_locks: [xxx]`），角色在對話中就會失去特色。這是需要請專家協助審視的核心區域。

### 6.3 主題與焦點的有效性

目前 `對話主題` 和 `討論焦點` 直接傳入 user message，未做任何前處理。當主題過於寬泛（如「聊聊 AI」）或焦點與角色 soul 的專業領域差距太大時，對話容易流於表面。可考慮：
- 讓 LLM 先根據兩個角色的 soul 重新詮釋主題，找到最能激發衝突或共鳴的切入點
- 焦點改為多點列舉，讓對話在中段有具體的「錨點」可以碰撞

### 6.4 對話碰撞度不足

目前規則說「認同就認同，不認同就直說」，但沒有給 LLM 足夠的動機去製造真正的觀點碰撞。若兩個角色的 soul 沒有明顯的立場差異，LLM 傾向讓他們互相補充而非對立。可考慮：
- 在 system prompt 中加入「找出兩個角色在這個主題上最可能不同意的那個點，讓它浮現出來」
- 或由使用者指定「哪個角色傾向正方，哪個傾向反方」

### 6.5 破音字表尚未完整

目前詞彙 Map 只有 5 個詞。台灣繁體在科技、法律、日常用語等領域有大量 MiniMax 讀音不準的詞彙，需要系統性整理後補入。

## 八、API 端點

| 路由 | 方法 | 說明 |
|------|------|------|
| `/api/convert/podcast/generate-script` | POST | Phase 1：腳本生成 |
| `/api/convert/podcast/generate-audio` | POST | Phase 2：標記 + TTS + 合併 |
| `/api/tts` | POST | 單句 TTS（含正規化），供其他功能共用 |

**Phase 1 Request**：
```json
{
  "characterIds": ["char_a_id", "char_b_id"],
  "topic": "AI 怎麼改變設計師的工作方式",
  "focus": "聚焦在工具替代與創意主導權的拉鋸",
  "wordCount": 600
}
```

**Phase 1 Response**：
```json
{
  "taskId": "xxx",
  "script": [
    { "speaker": "角色A", "characterId": "char_a_id", "text": "台詞內容" },
    { "speaker": "角色B", "characterId": "char_b_id", "text": "台詞內容" }
  ]
}
```

**Phase 2 Request**：
```json
{
  "taskId": "xxx",
  "script": [...]   // 選填，若使用者有編輯腳本則帶入
}
```

**Phase 2 Response**：
```json
{
  "audioUrl": "https://storage.googleapis.com/ailivex-2026-assets/podcast/xxx.mp3"
}
```
